## Checklist

### 1. Evidence of Need

**Applies to:** Both | **Ref:** P4133 3.1 / 4.1 | **Failure modes:** A2

**Score:** 1 / 2
**Notes:**
Section 1.1 (Motivation) identifies a genuine gap: C++ has no standard
vocabulary or framework for asynchronous execution despite asynchrony
appearing in every domain. The paper argues that `std::async` /
`std::future` / `std::promise` is "inefficient, hard to use correctly,
and severely lacking in genericity," and that C++17 parallel algorithms
are synchronous and not composable. The implicit standardization
argument is that only a standard vocabulary type can unify the
incompatible async models that every library and framework currently
invents independently. However, the paper does not contain a structured
cost/benefit analysis answering the three P4133 questions: what problem
the ecosystem cannot solve (e.g., why stdexec on vcpkg is
insufficient), what property of standardization is required (portability,
guaranteed availability, ABI stability, teaching curriculum), and why the
benefit exceeds the cost of implementer burden, ABI commitment, and
perpetual maintenance for a large, complex API surface.

---

### 2. Implementation

**Applies to:** Both | **Ref:** P4133 3.2 / 4.2 | **Failure modes:** A1, C2

**Score:** 2 / 2
**Notes:**
NVIDIA's stdexec is the official reference implementation, actively
maintained on GitHub with over 2200 stars and 80+ contributors. It
includes comprehensive unit tests, benchmarks, documentation, and
CUDA GPU extensions. Meta's libunifex is an earlier independent
prototype implementing a previous revision of the design. The paper
itself contains extensive real-world code examples spanning Windows
IOCP networking, CUDA GPU kernel launches, HTTP server pipelines,
thread pool scheduling, coroutine integration, and custom scheduler
implementations. Multiple production deployments at NVIDIA and Meta
have validated the design in practice. The combination of a mature,
publicly available reference implementation with tests and
documentation, an independent second implementation, and production
deployment meets the rubric threshold for a complete library.

### 4. Steel Man Against Standardization / Adoption

**Applies to:** Both | **Ref:** P4133 3.3 / 4.4 | **Failure modes:** D2

**Score:** 1 / 2
**Notes:**
The paper argues effectively that existing standard facilities
(`std::async`, `std::future`, parallel algorithms) are insufficient
for modern asynchronous programming. The implicit case for
standardization - that a common vocabulary type is needed so that
libraries, frameworks, and execution resources can interoperate - is
embedded throughout the paper. However, the paper never presents and
confronts the strongest counter-argument: stdexec already exists as
a high-quality non-standard library that could be distributed via
vcpkg or Conan without the costs of standardization (ABI commitment,
implementer burden across all major standard library implementations,
perpetual maintenance of a large and complex API surface). The paper
also does not confront the argument that the complexity cost of the
design - deeply template-heavy, concept-constrained, with a large
number of customization points - may exceed the benefit for many
users who would prefer simpler async models.

---

### 5. Steel Man of Competing Designs

**Applies to:** Both | **Ref:** P4133 3.4 / 4.5 | **Failure modes:** D1, D4

**Score:** 1 / 2
**Notes:**
The paper acknowledges P0443 (A Unified Executors Proposal for C++)
as the foundation and states it evolved from that design and its
companion papers. The design evolution across ten revisions reflects
substantial refinement. However, the paper does not contain a
dedicated section presenting competing designs at their strongest.
Missing steel men include: Boost.Asio's completion handler / async
result model (widely deployed, simpler mental model), pure
coroutine-based approaches (e.g., libcoro, cppcoro - no
sender/receiver abstraction), redesigned future/promise models
(e.g., continuable futures as in Seastar or folly::Future), and the
"do nothing" option (leave async to the ecosystem). Each alternative
has genuine advantages (simplicity, existing deployment, smaller API
surface) that the paper does not acknowledge or defeat.

### 8. Decision Record

**Applies to:** Both | **Ref:** P4133 3.7 | **Failure modes:** B1, B2, E1

**Score:** 1 / 2
**Notes:**
The paper embeds design rationale throughout: the choice of
schedulers/senders/receivers as core abstractions, the
customization-point mechanism via domains, the pipe syntax for
composition, the tripartite completion signals (value, error,
stopped), and the relationship between senders and coroutine
awaitables. The ten-revision history reflects extensive design
iteration. However, the paper lacks a formal decision record
documenting: why senders/receivers were chosen over completion
handlers, futures, or coroutines as the primary abstraction;
alternatives considered and explicitly rejected at each major
design fork; dissenting views from SG1 or LEWG review; trade-offs
consciously accepted (e.g., compile-time cost of deeply nested
template types, difficulty of debugging sender chains, the learning
curve for the receiver concept); and conditions under which the
committee should revisit fundamental design choices.

### 9. Domain Coverage Attestation

**Applies to:** Both | **Ref:** P4133 3.8 | **Failure modes:** A3, D3, G1, G2

**Score:** 1 / 2
**Notes:**
Section 1.1 names multiple domains: scientific computing, video
games, financial services, mobile devices, GPUs, supercomputers.
The author list spans NVIDIA (GPU/HPC), Meta (infrastructure), and
independent contributors. Code examples demonstrate Windows IOCP
networking, CUDA GPU scheduling, HTTP server pipelines, and thread

represented in design review, how they were consulted, or which
domains remain underrepresented. Notable gaps: embedded/real-time
systems (where heap allocation constraints and exception-freedom

systems with different constraints), and financial services (where
deterministic latency requirements may conflict with the design's
flexibility).

## Executive Summary

P2300R10 proposes `std::execution`, a framework for managing
asynchronous execution on generic execution resources, built around
schedulers, senders, and receivers. The proposal is the culmination of
over a decade of work on C++ executors, has a mature reference
implementation (NVIDIA's stdexec), an independent prototype (Meta's
libunifex), production deployments, and was adopted into C++26 in June
2024. The technical design is sophisticated and well-specified, with
complete proposed wording and extensive real-world code examples.

However, the paper was written before P4133 and does not contain the
evidence framework that rubric requires. The P4133 gaps are process
gaps, not quality gaps - the absence of formal cost/benefit analysis,
steel-man arguments, post-adoption metrics, retrospective triggers,
and a prediction registry does not diminish the technical achievement
but does mean the committee adopted the largest library addition in
recent C++ history without the structured evidence framework P4133
calls for.

**Overall readiness:** Technically mature and already adopted into
C++26. The P4133 evidence gaps are retrospective observations, not
blocking deficiencies - the ship has sailed. Future papers building on
P2300 (e.g., async scope, system scheduler, networking) should address
these rubric items proactively.

**Strongest areas** (items scoring 2): Implementation (item 2) - the
reference implementation is production-grade with multiple independent
implementations and deployment evidence.

**Critical gaps** (items scoring 0): Post-adoption metrics (item 6),
retrospective trigger (item 7), prediction registry (item 10). These
three items represent the absence of any structured commitment to
evaluate whether the design achieves its goals after adoption.

## Improvement Tips

### 1. Evidence of Need (P4133 3.1)

**Scored 1.** Add a structured cost/benefit analysis answering three
questions explicitly. Why the ecosystem cannot solve this problem with
stdexec distributed via vcpkg (answer: vocabulary types must be in the
standard to enable library interoperability - an HTTP library and a
database library need to compose their async operations using a shared
model). What property of standardization is required (answer: a
guaranteed-available vocabulary type that every library can depend on
without adding a third-party dependency). Why the benefit exceeds the
cost (answer: quantify the cost of incompatible async models across
the ecosystem and compare it to the cost of implementer burden for a
large, complex standard library addition).

### 2. Implementation (P4133 3.2)

**Scored 2.** No improvement needed. For future papers in the P2300
family, consider summarizing benchmark results and test coverage
metrics inline rather than requiring the reader to visit GitHub.

### 4. Steel Man Against Standardization (P4133 3.3)

**Scored 1.** Add a dedicated section confronting: "stdexec already
exists as a high-quality header-only library. Any project that needs
sender/receiver can add it via vcpkg today. Standardization commits
three major implementations to maintaining a large, complex API
surface in perpetuity, with ABI stability guarantees that may prevent
future improvements. The complexity cost (deep template nesting,
concept-heavy interfaces, large number of customization points) may
outweigh the benefit for the majority of C++ developers who need
simple async patterns." Confront this with evidence: the fraction of
C++ projects that use package managers, the historical failure of
ecosystem-only vocabulary types to achieve universal adoption, and
the specific interoperability benefits that require standardization.

### 5. Steel Man of Competing Designs (P4133 3.4)

**Scored 1.** Present each competing model at its strongest.
Boost.Asio's completion handler model: simpler, deployed for 20+
years, covers networking natively, lower learning curve. Pure
coroutine approaches: simpler mental model for sequential-looking
async code, no receiver concept to learn, direct language support.
Redesigned futures (Seastar, folly::Future): familiar abstraction,
continuations without the sender/receiver formalism. Do nothing:
the ecosystem is already solving async without a standard framework.
For each, document its advantages honestly before explaining why
sender/receiver is the better standard choice.

### 6. Post-Adoption Metrics (P4133 3.5)

**Scored 0.** Define measurable success criteria. Examples: all
three major implementations (libstdc++, libc++, MSVC STL) ship a
conforming implementation within two releases of C++26; at least
three major open-source libraries adopt `std::execution` as their
async backend within five years; developer surveys show sender/
receiver comprehension rates above a defined threshold; the number
of incompatible async models in the ecosystem decreases measurably.

### 7. Retrospective Trigger (P4133 3.6)

**Scored 0.** Commit to a look-back at C++29 or six years after
C++26, whichever comes first. Define the questions: did all major
implementations ship? Did the ecosystem adopt sender/receiver as
the default async vocabulary? Did the customization mechanism prove
usable? Were the compile-time costs acceptable? Did the teaching
story work? What design limitations emerged in practice?

### 8. Decision Record (P4133 3.7)

**Scored 1.** Add a formal record documenting: why
sender/receiver over completion handlers (Asio model), futures,
or raw coroutines; what was traded away (simplicity, smaller API
surface, lower learning curve); dissenting views from SG1/LEWG
review (e.g., concerns about complexity, compile times, debuggability);
conditions for revisiting (e.g., if implementations cannot ship
within two releases, if adoption surveys show the abstraction is
too complex for target users).

### 9. Domain Coverage Attestation (P4133 3.8)

**Scored 1.** Systematize the coverage. List every affected domain
(GPU/HPC, networking, embedded, game engines, financial services,
server infrastructure, mobile, desktop GUI) and state which domains
provided input during design review. Acknowledge gaps - embedded/
real-time and game engine domains have specific constraints
(no heap allocation, custom job systems, deterministic latency)
that may not be well-served by the current design. Document plans
to fill those gaps.

### 10. Prediction Registry (P4133 3.9)

**Scored 0.** Record the paper's implicit predictions with
falsifiable criteria. "Sender/receiver will become the ecosystem's
default async vocabulary within five years of C++26" (testable via
ecosystem surveys). "All three major implementations will ship
conforming implementations within two releases" (testable by
release date). "Compile-time overhead of sender chains will be
competitive with hand-written async code" (testable by benchmark
comparison). "The customization mechanism will prove sufficient
for GPU, networking, and file I/O execution resources" (testable
by implementation attempts in each domain). Each claim needs a
revisit date.
